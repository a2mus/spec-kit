# Research: Clean Fork for Upstream Sync

**Feature Branch**: `002-clean-runtime-artifacts` | **Date**: 2026-06-22

## Research Summary

This feature is a repository-hygiene operation, not a code-change feature. The
research resolves the unknowns identified during technical-context analysis and
confirms the approach is safe, reversible, and compatible with upstream merges.

---

## R-001: Upstream Tracking Behaviour

**Decision**: The upstream `github/spec-kit` repository does NOT track any of the
directories this fork currently tracks as runtime artifacts.

**Rationale**: `git ls-tree` against `upstream/main` confirms:

- `.agents/` — **not tracked upstream** (0 files)
- `.claude/` — **not tracked upstream** (0 files)
- `.kilocode/`, `.opencode/` — **not tracked upstream** (0 files)
- `.specify/` — upstream tracks only **1 file**: `.specify/memory/constitution.md`
  (the template)
- `CLAUDE.md` — **not tracked upstream**
- `AGENTS.md` — **tracked upstream** (shared context file, also tracked in fork)

The upstream `.gitignore` already includes blanket ignores for `.specify/` and
`specs/`, plus agent-specific directories like `.github/agents/`,
`.github/prompts/`, and `.github/copilot-instructions.md`.

**Alternatives considered**: None — the upstream behaviour is a fact, not a
design choice. The fork's current tracking is the anomaly.

## R-002: Safe `git rm --cached` Behaviour

**Decision**: Use `git rm --cached` to remove files from the index without
deleting them from the working directory.

**Rationale**: The `--cached` flag removes paths from version control only. The
files remain on disk, so the maintainer's local environment keeps working
without re-running `specify init`. This is confirmed by git documentation and is
the standard operation for untracking previously committed files.

**Risk**: After `git rm --cached`, the files appear as "untracked" in
`git status` until `.gitignore` rules cover them. This is expected and harmless
once the `.gitignore` update is committed in the same operation.

**Alternatives considered**:
- `git update-index --assume-unchanged` — rejected because it only affects the
  local clone, not the repository state. Other clones would still track the files.

## R-003: `.gitignore` Pattern Strategy

**Decision**: Use directory-level ignore rules that match the upstream's
conventions where possible, plus fork-specific rules for directories the
upstream does not reference.

**Rationale**: The upstream `.gitignore` already ignores `.specify/` and `specs/`
wholesale. Our fork should add rules for:

1. **Agent integration directories** (runtime copies of skills/workflows):
   `.agents/`, `.claude/`, `.kilocode/`, `.opencode/`
2. **Agent context files** that are generated per-integration:
   `CLAUDE.md` (but NOT `AGENTS.md`, which is the upstream-tracked project doc)
3. **`.specify/` subdirectories** that hold runtime artifacts: `scripts/`,
   `templates/`, `memory/`, `integrations/`, `workflows/`, `feature.json`
4. **Negation rules** (`!`) to keep the three project config files tracked:
   `!.specify/extensions.yml`, `!.specify/init-options.json`,
   `!.specify/integration.json`
5. **Negation rules** for the bundled `agent-context` extension (canonical source
   in `extensions/agent-context/` — separate from runtime installs in
   `.specify/extensions/`):  already tracked and should remain tracked.

**Pattern**: Use a dedicated `# Fork-specific: runtime artifacts` section to keep
these rules grouped and documented.

**Alternatives considered**:
- Ignoring `.specify/` wholesale (like upstream) — rejected because our fork
  needs the three project config files tracked. Selective ignores with negation
  rules are more appropriate.
- Per-file ignore rules instead of directory rules — rejected as overly
  brittle; directory-level rules cover future additions automatically.

## R-004: Preserving the Ponytail Extension

**Decision**: The ponytail extension lives under `extensions/ponytail/` (the
canonical bundled-extensions directory), NOT under `.specify/extensions/`. It is
already properly tracked and requires zero changes.

**Rationale**: `git ls-files extensions/ponytail/` confirms 6 tracked files.
The `.specify/extensions/` directory is where spec-kit *installs* extensions at
runtime — the canonical sources live in `extensions/`. The ponytail extension is
unaffected by ignoring `.specify/extensions/` runtime artifacts.

**Alternatives considered**: None — no conflict exists.

## R-005: CLAUDE.md Tracking Status

**Decision**: `CLAUDE.md` should be untracked. It is a generated context file
with no canonical content (currently just the SPECKIT START/END markers).

**Rationale**: The file is 5 lines of auto-generated marker content. It is
regenerated by `specify init` or `speckit.agent-context.update`. Upstream does
not track it. Keeping it tracked creates merge noise when integrations change.

`AGENTS.md`, by contrast, **remains tracked** — it is the project documentation
file that upstream also tracks, and contains substantial authored content.

**Alternatives considered**:
- Tracking `CLAUDE.md` as project config — rejected because it has no authored
  content; its value is entirely derived from the running integration state.

## R-007: `specify init` Re-Entrant Behaviour

**Decision**: Confirm and document that `specify init --here` on a clone with
existing project config files preserves those files.

**Rationale**: The spec's Assumption #3 states this. The `setup-plan.ps1` script
already demonstrates re-entrant logic (it checks `Test-Path` before copying).
The `specify init` command uses `--force` to merge into non-empty directories
and the `--ignore-agent-tools` flag when the agent CLI isn't installed.

A fresh clone would run:
```powershell
specify init --here --integration opencode --ignore-agent-tools
```

This regenerates all runtime artifacts (scripts, templates, skills, workflows)
while leaving the three project config files and `extensions.yml` intact.

**Risk**: If a future version of `specify init` changes behaviour, the setup
guide should be updated. The verification step in the guide covers this.

**Alternatives considered**: None — this is the standard flow.

## R-008: Rollback Strategy

**Decision**: Create a git tag at the pre-cleanup commit so the exact state can
be restored with `git checkout <tag>`.

**Rationale**: A tag is simpler than a branch for a restore point — it's
immutable and discoverable with `git tag -l`. The rollback command would be:

```bash
git checkout pre-cleanup-002 -- .
git checkout HEAD -- .gitignore  # keep the updated ignore rules
```

Or, for a full revert to the pre-cleanup tracked-file set:
```bash
git revert <cleanup-commit-sha>
```

**Alternatives considered**:
- Stashing — rejected because it's ephemeral and per-developer.
- Branch — workable but a tag is lighter and more conventional for restore
  points.

---

## R-009: Post-Cleanup Merge Compatibility

**Decision**: After cleanup, upstream merges will be cleaner because the fork no
longer tracks files that upstream ignores.

**Rationale**: The 298-file diff between the fork and upstream on these paths
drops to near-zero. The only potential merge-conflict surface is `AGENTS.md` (a
file both sides track and both sides may edit). This is a genuine content merge
and is expected to be rare and resolvable.

**Alternatives considered**: None — this is the primary goal of the feature.
