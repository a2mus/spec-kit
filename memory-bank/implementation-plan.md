# Implementation Plan: Clean Fork for Upstream Sync

## Overview

Clean the a2mus/spec-kit fork by removing runtime artifacts from git tracking. This enables frictionless upstream merges from `github/spec-kit` while preserving the ponytail extension (already properly structured) and project-specific configuration.

## Critical Findings (Audit Results)

The original plan assumed 49 gstack skills + 6 PowerShell scripts were "customizations" to migrate into an extension. A thorough audit revealed this assumption is **incorrect**. Here's what actually exists:

### Finding 1: GStack Skills Are External, Not Spec-Kit Customizations

The 49 gstack skills (e.g., `gstack-autoplan`, `gstack-review`, `gstack-ship`) are from the **GStack framework** — a separate AI coding assistant project by Garry. Evidence:

- SKILL.md files reference `~/.claude/skills/gstack/bin/*` binaries
- They use GStack telemetry (`~/.gstack/analytics/`), GStack config (`gstack-config`), and GStack-specific conventions (Boil the Lake, Eureka logging, Codex dual-voice)
- They have no relationship to Spec Kit's extension system, templates, or commands
- They were installed into `.agents/skills/gstack-*` by the gstack installer and accidentally committed

**Conclusion**: GStack skills should NOT be packaged as a spec-kit extension. They belong to the gstack project and should be managed by gstack's own installer per-developer.

### Finding 2: No Custom PowerShell Scripts Exist

The 6 PowerShell scripts in `.specify/scripts/powershell/` are **runtime copies** of upstream scripts, not custom files:

| `.specify/scripts/powershell/` file | Source |
|---|---|
| `common.ps1` | Copied from `scripts/powershell/common.ps1` during `specify init` |
| `create-new-feature.ps1` | Copied from `scripts/powershell/create-new-feature.ps1` during `specify init` |
| `setup-plan.ps1` | Copied from `scripts/powershell/setup-plan.ps1` during `specify init` |
| `setup-tasks.ps1` | Copied from `scripts/powershell/setup-tasks.ps1` during `specify init` |
| `check-prerequisites.ps1` | Copied from `scripts/powershell/check-prerequisites.ps1` during `specify init` |
| `update-agent-context.ps1` | Installed by the `agent-context` extension during `specify extension add` |

The upstream scripts at `scripts/powershell/` (5 files) are already tracked and part of the core package. The `.specify/` copies are runtime artifacts.

**Conclusion**: No custom PowerShell scripts need migration. The `.specify/scripts/` directory should not be tracked.

### Finding 3: The Real Problem Is Git-Tracked Runtime Artifacts

| Directory | Files Tracked | Should Be Tracked? |
|---|---|---|
| `.agents/skills/` | 178 files (98 gstack + 80 spec-kit) | **No** — runtime-generated by `specify init` |
| `.claude/skills/` | 69 files | **No** — runtime-generated by Claude integration |
| `.specify/scripts/` | 6 files | **No** — runtime copies of upstream/extension scripts |
| `.specify/templates/` | 38 files | **Mostly no** — runtime copies; see exceptions below |
| `.specify/extensions/` | 9 files | **Mostly no** — runtime installed extensions; see exceptions below |
| `.specify/workflows/` | 2 files | **Mostly no** — runtime; see exceptions below |
| `.specify/memory/` | 1 file | **Yes** — project constitution (but is still a template) |
| `.specify/init-options.json` | 1 file | **Yes** — project init configuration |
| `.specify/integration.json` | 1 file | **Yes** — active integration config |
| `.specify/extensions.yml` | 1 file | **Yes** — installed extensions manifest |
| `extensions/ponytail/` | 6 files | **Yes** — custom extension, already properly structured |

**Total runtime artifacts to untrack**: ~298 files across 5 directories.

### Finding 4: Ponytail Extension Is Already Correct

The `extensions/ponytail/` extension is properly structured with valid `extension.yml`, commands, config template, hooks, and bundled in `pyproject.toml`. It requires no changes.

### Finding 5: No Actual Customizations Need Extension Packaging

After auditing every file, there are **zero custom scripts or skills** that need to be packaged into a new extension. The only genuine customization is the ponytail extension, which is already done correctly.

## Architecture Decisions

### AD-1: .gitignore Over Extension

**Decision**: Use `.gitignore` to exclude runtime artifacts rather than creating a new extension.
**Rationale**:
- There are zero custom scripts or skills to package — an extension would contain nothing
- The `.specify/`, `.agents/`, and `.claude/` directories are explicitly designed as runtime/project-local directories
- The upstream spec-kit project already expects these to be gitignored (per their conventions)
- Adding them to `.gitignore` is the minimal, correct solution

### AD-2: GStack Skills Are Out of Scope

**Decision**: Do not include gstack skills in any spec-kit extension or git tracking.
**Rationale**:
- GStack is a separate project with its own installer, versioning, and update mechanism
- Packaging gstack as a spec-kit extension would create a phantom dependency
- Each developer should install gstack separately via its own setup process
- The gstack skills in `.agents/skills/gstack-*` were accidentally committed and should be removed

### AD-3: Preserve Project Config, Not Runtime Copies

**Decision**: Keep only the 3 project config files (`.specify/extensions.yml`, `init-options.json`, `integration.json`) in git. Remove all runtime copies.
**Rationale**:
- Config files define the project's spec-kit setup and are valuable to track
- Runtime copies (scripts, templates, extensions) are re-generated by `specify init` and `specify extension add`
- Tracking runtime copies creates merge conflicts with upstream without any benefit
- The upstream scripts at `scripts/powershell/` are already tracked and serve as the source of truth

### AD-4: Constitution Template Not Yet Customized

**Decision**: Do NOT track `.specify/memory/constitution.md` — it's still the upstream template.
**Rationale**:
- Reading the file confirms it's the unmodified constitution template
- Once customized for this project, it should be committed as a project-specific file
- For now, it's a runtime artifact that will be re-generated by `specify constitution`

## Edge Cases & Error Handling

### EC-1: specify init Overwrites Project Config
**Risk**: Running `specify init` after cloning could overwrite `.specify/init-options.json` or `.specify/integration.json`.
**Mitigation**:
- These config files ARE tracked in git, so `specify init` will see them as existing project config
- Spec Kit's init flow preserves existing configuration when `.specify/` already has config files
- Test this explicitly in Phase 3 verification

### EC-2: GStack Users Lose Skills After Clone
**Risk**: Developers who use gstack will not have gstack skills after cloning (since `.agents/skills/gstack-*` is gitignored).
**Mitigation**:
- Document in `docs/DEVELOPER-SETUP.md` that gstack must be installed separately
- gstack has its own installer (`~/.claude/skills/gstack/setup --team` or equivalent)
- This is the correct behavior — gstack should NOT be distributed via spec-kit

### EC-3: Agent-Context Extension Hooks Stop Firing
**Risk**: After untracking `.specify/extensions/agent-context/`, the extension hooks may not fire.
**Mitigation**:
- The extension is still installed on disk (only removed from git index)
- The `.registry` file tracks installed extensions and persists across git operations
- `specify extension list` will still show agent-context as installed
- If needed, re-install: `specify extension add agent-context`

### EC-4: Upstream Adds New Tracked Files to .specify/
**Risk**: Future upstream changes could add files to `.specify/` subdirectories that we now gitignore.
**Mitigation**:
- Upstream does NOT track `.specify/` files — they are project-local by design
- Our `.gitignore` matches upstream conventions
- The 3 config files we do track (`extensions.yml`, `init-options.json`, `integration.json`) are unlikely to conflict with upstream since upstream doesn't generate them

### EC-5: Memory-Bank Directory Has Valuable Planning Content
**Risk**: `memory-bank/` is already untracked but contains this implementation plan.
**Mitigation**:
- `memory-bank/` is already correctly gitignored (not tracked)
- This plan can be preserved by committing it to a different location (e.g., `docs/plans/cleanup-runtime-artifacts.md`) or keeping it as a local artifact
- The plan's value is in its execution, not its permanent storage

## Rollback Strategy

If the cleanup causes problems:

```bash
# Option 1: Reset to backup commit (instant, complete rollback)
git reset --hard pre-cleanup-v0.10.3

# Option 2: Revert the cleanup commit (preserves history)
git revert HEAD

# Option 3: Manual — remove .gitignore entries and re-add files
git rm --cached .gitignore
# Re-add specific directories
git add .agents/ .claude/ .specify/
git commit -m "rollback: restore runtime artifact tracking"
```

## Upstream Merge Workflow (Post-Cleanup)

After this cleanup, the fork maintenance workflow becomes:

```bash
# 1. Fetch upstream changes
git fetch upstream

# 2. Merge into fork's main branch
git merge upstream/main

# 3. Resolve conflicts (expected: near-zero)
# Only possible conflict: .gitignore (upstream may add new entries)
# Resolution: keep both sets of entries

# 4. Push to fork
git push origin main
```

**Expected conflict surface**: Only `.gitignore` — and even that is unlikely since our additions follow upstream conventions.
